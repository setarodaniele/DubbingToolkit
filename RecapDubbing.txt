Installation/
La cartella Installation contiene installazioni locali complete di Python gestite dal progetto.
Il sistema utilizza esclusivamente queste versioni come runtime, evitando dipendenze da Python installati nel sistema operativo.
Il Launcher e VenvManager selezionano e utilizzano queste installazioni per:
- Creare e gestire gli ambienti virtuali (venv/)
- Garantire coerenza tra ambienti diversi
- Rendere il progetto portabile e replicabile
Gli ambienti virtuali vengono sempre costruiti a partire da queste installazioni locali.

* Tutti i file presenti nella cartella `credentials/` contengono chiavi e credenziali personali per i servizi TTS e modelli.
Questi file NON devono essere inclusi nel repository e devono essere gestiti localmente.
I file `.template.json` fungono da riferimento per la struttura richiesta e devono essere utilizzati come base per la configurazione.

= SEZIONE 2 – Convenzioni di progetto =
=======================================

Convenzioni e Naming
--------------------

Stato del progetto
- Il progetto è attualmente in uno stato ibrido:
  una parte dei moduli e delle cartelle esistenti non segue le convenzioni definite di seguito.
- Per preservare la stabilità, il codice esistente NON viene modificato automaticamente.
- Le convenzioni si applicano obbligatoriamente a tutti i nuovi file e moduli.
- Un eventuale refactoring completo per allineare il codice esistente alle convenzioni verrà pianificato separatamente,
  solo dopo la stabilizzazione della pipeline principale.

Regole generali
- Tutti i percorsi utilizzati negli script devono essere relativi alla root del progetto.
- Nessuna stringa hardcoded nei moduli runtime:
  tutti i messaggi devono provenire dai file JSON di localizzazione,
  ad eccezione del bootstrap iniziale e degli script di servizio non destinati al runtime.
- Gli script di servizio sono esclusi dal concetto di runtime applicativo e possono contenere logica tecnica non localizzata.
- In caso di chiavi mancanti nei file JSON, i sistemi devono evidenziare il problema
  tramite fallback esplicito (es. [MISSING: key] o equivalente).
- Le chiavi mancanti non sono ammesse in ambiente stabile e devono essere corrette.
- Tutti gli script devono seguire la struttura definita nella sezione "Struttura degli script".
- I moduli non ancora allineati a questo comportamento devono essere adeguati
  durante le attività di refactoring.

Naming file e cartelle
- Nuove cartelle: minuscolo con underscore (no spazi, no caratteri speciali)
- Moduli Python: minuscolo con underscore (es. tts_google.py)
- Script temporanei: prefisso obbligatorio test_

Naming codice
- Classi → CamelCase (es. TextToSpeech)
- Funzioni e variabili → minuscolo con underscore (es. sintetizza_voce)

Script e moduli
------------------------------------------------------------
- Script operativo → file .py eseguibile direttamente (runtime o utility)
- Modulo → file .py importabile all’interno di altri script

Stile codice
- Riferimento generale: PEP8
- Indentazione: 4 spazi
- Lunghezza righe: flessibile se necessario per leggibilità operativa

Struttura degli script
------------------------------------------------------------
- Tutti gli script del progetto devono seguire una struttura standardizzata e obbligatoria per garantire leggibilità,
  manutenzione e coerenza tra moduli Python e PowerShell.

Regole strutturali generali
- Ogni script deve essere organizzato in blocchi numerati progressivi (1, 2, 3, ...).
- La numerazione deve essere continua, ordinata e senza salti o duplicazioni.
- Il numero totale dei blocchi non è vincolato: uno script può contenere un numero variabile di blocchi (5, 10, 20 o più) in base alla complessità.
- I blocchi definiti nel template rappresentano una struttura di riferimento e non una numerazione rigida.
- Ogni blocco principale deve avere:
  - numero identificativo
  - titolo descrittivo breve in inglese
- Ogni blocco può contenere sottoblocchi numerati (es. 1.1, 1.2, 2.1).
- Ogni sottoblocco deve includere una micro-intestazione descrittiva su singola riga.
- Le sezioni logiche fondamentali (imports, configurazione, utilities, logica, esecuzione) devono essere sempre presenti,
  anche se suddivise in più blocchi.
- La numerazione dei blocchi deve essere coerente con il flusso logico dello script.

Intestazione script (obbligatoria)
- Ogni script deve iniziare con una intestazione standard conforme al template ufficiale.
- L’intestazione deve includere:
  - nome dello script
  - breve descrizione funzionale (1–3 righe)
  - input principali (se presenti)
  - output principali (se presenti)
  - note operative essenziali
- L’intestazione deve essere scritta in inglese, come già indicato, e non contenere logica esecutiva.

Regole di organizzazione interna
- Imports e dipendenze devono essere isolati nel primo blocco.
- Configurazioni e path devono essere separati in uno o più blocchi dedicati.
- Le utility devono essere raggruppate in blocchi autonomi.
- La logica principale deve essere separata dalla fase di esecuzione (main flow).
- Il blocco di esecuzione principale (main) deve essere sempre posizionato alla fine dello script
  e deve corrispondere all’ultimo blocco numerato.
- L’ordine dei blocchi deve seguire il flusso logico dello script.

Standard di funzioni
- Ogni funzione deve includere una descrizione inline su singola riga.
- La descrizione deve essere breve e in inglese.
- Il formato deve essere:
  nome_funzione – descrizione breve (one line)
- Le funzioni devono essere logicamente raggruppate per area funzionale.

Riferimento ufficiale
- Tutti gli script devono seguire il template definito in:
  RecapDubbing.txt – Sezione 6
- La Sezione 6 di questo documento rappresenta il modello strutturale ufficiale
  e l'implementazione canonica delle regole definite in questa sezione.
  
Gestione messaggistica (Python + Runtime system)
------------------------------------------------------------
- Tutti i messaggi di interfaccia Python devono essere gestiti tramite:
  core/messages.py → classe Messages
- Il sistema centralizza:
  - caricamento lingua da settings.json (interface_lang)
  - lettura file JSON da locale/Active/
  - In caso di chiavi mancanti nei file JSON, il sistema utilizza fallback esplicito [MISSING: key].
- Lo script principale e tutti i moduli runtime devono utilizzare esclusivamente questo sistema.
- Eccezione:
  script di bootstrap iniziale e script di servizio possono usare output non localizzato.

Gestione lingue (locale)
------------------------------------------------------------
- La gestione delle lingue è centralizzata nella cartella locale/
- locale/Active/
  contiene solo lingue localizzate e attive nel sistema
  → usate per interfaccia e runtime
- locale/System/
  contiene metadati e definizioni di sistema:
  - languages.json → lingue concettualmente supportate
  - whisper_languages.json → lingue effettivamente supportate da Whisper
- Le lingue attive sono l’unico set valido per menu e selezione utente.

Gestione messaggistica PowerShell
------------------------------------------------------------
- Sistema basato su moduli dedicati:
  ps/Messages.psm1
  ps/Logging.psm1
- Messages.psm1:
  - gestione messaggi tramite chiavi
  - supporto Hashtable e JSON object
  - interpolazione parametri
  - fallback: "Missing key: <key>"
- Logging.psm1:
  - Write-Log
  - livelli: INFO / WARN / ERROR
  - integrazione con Get-Message
  - output console colorato per livello
- Il sistema PowerShell è equivalente concettualmente al sistema Python,
  ma separato per ambiente runtime.
- I moduli PowerShell devono utilizzare esclusivamente questo sistema.
- I moduli legacy non ancora allineati sono considerati compatibilità temporanea.

Script di test
------------------------------------------------------------
- Tutti gli script di test devono avere prefisso obbligatorio:
  test_
- Il prefisso deve essere rimosso quando lo script entra nel flusso operativo.
- Non sono ammessi nomi alternativi.

Compatibilità legacy
------------------------------------------------------------
- Il codice esistente che non rispetta le convenzioni viene mantenuto invariato.
- Le nuove regole NON devono rompere la compatibilità con il codice già funzionante.
- Il refactoring completo del codice legacy verrà pianificato separatamente,
  solo dopo la stabilizzazione della pipeline principale.

Code Documentation Rules
------------------------------------------------------------
- I commenti nel codice devono essere scritti in inglese.
- Le stringhe dell’interfaccia utente devono essere gestite tramite sistema di localizzazione.
- Durante lo sviluppo si utilizza italiano come base (locale/Active/it.json),
  ma il codice deve rimanere documentato in inglese.
- Le stringhe di interfaccia possono essere sviluppate in italiano come base locale,
  ma devono essere sempre strutturate per successiva internazionalizzazione.
- L’obiettivo è mantenere un codice internazionalmente leggibile indipendentemente dalla lingua dell’interfaccia.

Gestione File di Lavoro (modello Workspace)
------------------------------------------------------------
- Tutti i file generati dalla pipeline vivono nel workspace attivo (Workspace/projects/<nome>/).
- L'app richiede sempre un progetto nominato aperto; non esiste workspace anonimo/predefinito.
- Ogni stage (audio_extraction, transcripts, translated, dubbed) è organizzato per track_id:
    Workspace/projects/<nome>/<stage>/<track_id>/current/   → output attivo
    Workspace/projects/<nome>/<stage>/<track_id>/archive/   → archivi precedenti (rotazione automatica)
- La rotazione viene eseguita da WorkspaceManager.rotate_to_archive() prima di ogni nuova generazione.
- Il pruning degli archivi è gestito da core/archive_pruner.py con anteprima dry-run e conferma utente.
  Politica di conferma:
    - stage dubbed: doppia conferma obbligatoria (audio finale irreversibile)
    - altri stage: singola conferma se show_confirmations=True in settings
- I file in video_input/ e audio_input/ del progetto non vengono mai modificati dalla pipeline.
- Tracciabilità affidata a project_info.json nella root del workspace attivo (aggiornato da info_manager.py).
  Il file viene creato automaticamente alla creazione o duplicazione del progetto.
  Ogni step della pipeline lo aggiorna tramite i metodi dedicati di InfoManager:
    - trascrivi_audio.py  → update_trascrizione(track_id, {...})
    - traduci_testo.py    → update_traduzione(track_id, {...})
    - tts_dubbing.py      → update_tts(track_id, {...})
  Include la sezione storico_rinomina per tracciare la cronologia delle rinominazioni del progetto.
  Il file project_info.txt (leggibile) viene rigenerato ad ogni aggiornamento tramite generate_info_file().
- Workspace/temp/ raccoglie file temporanei del workspace corrente.

Error Reporting

- Apertura client email tramite mailto: (nessuna credenziale SMTP richiesta).
- Email developer: SmallSoftwareHouse@gmail.com

Crash Detection vs Voluntary Close
------------------------------------
- Regista.py distingue tre percorsi di uscita:
    normal         → uscita dal menu (tasto X)
    window_closed  → chiusura X della finestra console (SetConsoleCtrlHandler)
    crash          → eccezione non gestita (Exception)
- exit_reason viene scritto nel log di sessione JSON da logger.close_session().
- Launcher.ps1 legge il log di sessione più recente al riavvio per decidere
  se mostrare il warning "crash rilevato" (solo se exit_reason non è normal/window_closed).

Logging PowerShell (sistema unificato)
---------------------------------------
- Tutti gli script PowerShell runtime usano ps/Logging.psm1 (Write-Log) + ps/Messages.psm1.
- Script allineati: VenvManager.ps1, Launcher.ps1, InstallDependencies.ps1.
- RuntimeDependencies.ps1 usa Write-RDLog proprio (con file logging aggiuntivo); allineamento futuro.

Pipeline (stage e percorsi Workspace)
--------------------------------------
| Step | Modulo               | Stage Workspace         |
|------|----------------------|-------------------------|
| 1 — Estrai audio    | estrai_tracce.py   | video_input → audio_extraction |
| 2 — Trascrivi       | trascrivi_audio.py | audio_extraction o audio_input → transcripts |
| 3 — Traduci         | traduci_testo.py   | transcripts → translated |
| 4 — Dubba (TTS)     | tts_menu.py        | translated → dubbed |

- Ogni stage usa la struttura <stage>/<track_id>/current/ per l'output attivo
  e <stage>/<track_id>/archive/ per archivi precedenti.
- WorkspaceManager (core/workspace_manager.py) gestisce il workspace attivo
  e la rotazione archivi. archive_pruner.py gestisce il pruning con conferma utente.
- Workspace attivo scritto in Workspace/active_workspace.json.

Workspace
----------
- L'app richiede sempre un progetto nominato aperto; nessun workspace predefinito anonimo.
- Progetti in Workspace/projects/<nome>/ (creabili/selezionabili da Regista.py menu Progetti).
- Ogni progetto ha la struttura stage completa (video_input, audio_input,
  audio_extraction, transcripts, translated, dubbed, output).
- Il workspace attivo viene risolto da WorkspaceManager.get_active() all'avvio.

Architettura Settings (implementata)
--------------------------------------
- I settings seguono un modello a tre livelli distinti:

  settings_default.json          → default di fabbrica dell'app (mai modificato a runtime)
  settings_persistent_default.json → default di fabbrica per le preferenze utente (mai modificato a runtime)
         ↓ (merge ad ogni avvio via BootstrapSettings.py)
  settings_persistent.json       → preferenze utente (modificate solo da settings_manager o installer)
         ↓
  settings.json                  → config runtime completa (file derivato, ricostruito ad ogni avvio)

- Tutte le scritture JSON usano write_json_atomic() (core/utils_settings.py): pattern .tmp + os.replace
  per prevenire corruzione in caso di crash o chiusura forzata.

- Lifecycle lingua interfaccia:
  1. Installer InnoSetup (opzionale) → scrive language_preset_from_installer=true in settings_persistent.json
  2. Launcher.ps1 → legge il flag; se presente salta il menu e mostra avviso; altrimenti menu interattivo
  3. BootstrapSettings.py → scrive interface_lang in settings_persistent.json e ricostruisce settings.json
  4. Runtime (menu 8→3) → modifica persistent + rebuild immediato
  5. Soft reset (menu 8→2) → ripristina settings_persistent.json da _default + rebuild
  6. Hard reset (menu 8→9) → stesso del soft + eliminazione e ricostruzione venv/

- Sync unidirezionale: Translation_Target_Lang → Dubbing_Lang
  (applicata in merge_and_rebuild e in menu_lingue.py al cambio lingua traduzione)

- I tre campi lingua sessione (Transcript_Audio_Spoken_Lang, Transcript_Target_Lang,
  Translation_Source_Lang) sono derivati da interface_lang ad ogni avvio e non persistono.

Reset
------
- Soft reset: ripristina preferenze utente ai default di fabbrica; non tocca venv/ né logs.
- Hard reset: stesso del soft + elimina venv/ (rename in venv_old poi delete) + rebuild al prossimo avvio.
  Richiede doppia conferma esplicita con descrizione delle conseguenze in tutte le 8 lingue.

Runtime Dependencies
---------------------
- ffmpeg 7.1.1: scaricato da GyanD/codexffmpeg GitHub Releases (formato .7z)
- voices_output: scaricato da SmallSoftwareHouse/DubbingToolkit GitHub Releases (formato .7z)
- Estrazione: Tools/7zr.exe (bundled nel repo e nel payload installer)
- Nessuna dipendenza da tool di sistema (tar.exe rimosso)
