Installation/
La cartella Installation contiene installazioni locali complete di Python gestite dal progetto.
Il sistema utilizza esclusivamente queste versioni come runtime, evitando dipendenze da Python installati nel sistema operativo.
Il Launcher e VenvManager selezionano e utilizzano queste installazioni per:
- Creare e gestire gli ambienti virtuali (venv/)
- Garantire coerenza tra ambienti diversi
- Rendere il progetto portabile e replicabile
Gli ambienti virtuali vengono sempre costruiti a partire da queste installazioni locali.

Questi file NON devono essere inclusi nel repository e devono essere gestiti localmente.
I file `.template.json` fungono da riferimento per la struttura richiesta e devono essere utilizzati come base per la configurazione.

Convenzioni e Naming
--------------------

Stato del progetto
- Il progetto è attualmente in uno stato ibrido:
  una parte dei moduli e delle cartelle esistenti non segue le convenzioni definite di seguito.
- Per preservare la stabilità, il codice esistente NON viene modificato automaticamente.
- Le convenzioni si applicano obbligatoriamente a tutti i nuovi file e moduli.
- Un eventuale refactoring completo per allineare il codice esistente alle convenzioni verrà pianificato separatamente,
  solo dopo la stabilizzazione della pipeline principale.

Regole generali
- Tutti i percorsi utilizzati negli script devono essere relativi alla root del progetto.
- Nessuna stringa hardcoded nei moduli runtime:
  tutti i messaggi devono provenire dai file JSON di localizzazione,
  ad eccezione del bootstrap iniziale e degli script di servizio non destinati al runtime.
- Gli script di servizio sono esclusi dal concetto di runtime applicativo e possono contenere logica tecnica non localizzata.
- In caso di chiavi mancanti nei file JSON, i sistemi devono evidenziare il problema
  tramite fallback esplicito (es. [MISSING: key] o equivalente).
- Le chiavi mancanti non sono ammesse in ambiente stabile e devono essere corrette.
- Tutti gli script devono seguire la struttura definita nella sezione "Struttura degli script".
- I moduli non ancora allineati a questo comportamento devono essere adeguati
  durante le attività di refactoring.

Naming file e cartelle
- Nuove cartelle: minuscolo con underscore (no spazi, no caratteri speciali)
- Moduli Python: minuscolo con underscore (es. tts_google.py)
- Script temporanei: prefisso obbligatorio test_

Naming codice
- Classi → CamelCase (es. TextToSpeech)
- Funzioni e variabili → minuscolo con underscore (es. sintetizza_voce)

Script e moduli
------------------------------------------------------------
- Script operativo → file .py eseguibile direttamente (runtime o utility)
- Modulo → file .py importabile all’interno di altri script

Stile codice
- Riferimento generale: PEP8
- Indentazione: 4 spazi
- Lunghezza righe: flessibile se necessario per leggibilità operativa

Struttura degli script
------------------------------------------------------------
- Tutti gli script del progetto devono seguire una struttura standardizzata e obbligatoria per garantire leggibilità,
  manutenzione e coerenza tra moduli Python e PowerShell.

Regole strutturali generali
- Ogni script deve essere organizzato in blocchi numerati progressivi (1, 2, 3, ...).
- La numerazione deve essere continua, ordinata e senza salti o duplicazioni.
- Il numero totale dei blocchi non è vincolato: uno script può contenere un numero variabile di blocchi (5, 10, 20 o più) in base alla complessità.
- I blocchi definiti nel template rappresentano una struttura di riferimento e non una numerazione rigida.
- Ogni blocco principale deve avere:
  - numero identificativo
  - titolo descrittivo breve in inglese
- Ogni blocco può contenere sottoblocchi numerati (es. 1.1, 1.2, 2.1).
- Ogni sottoblocco deve includere una micro-intestazione descrittiva su singola riga.
- Le sezioni logiche fondamentali (imports, configurazione, utilities, logica, esecuzione) devono essere sempre presenti,
  anche se suddivise in più blocchi.
- La numerazione dei blocchi deve essere coerente con il flusso logico dello script.

Intestazione script (obbligatoria)
- Ogni script deve iniziare con una intestazione standard conforme al template ufficiale.
- L’intestazione deve includere:
  - nome dello script
  - breve descrizione funzionale (1–3 righe)
  - input principali (se presenti)
  - output principali (se presenti)
  - note operative essenziali
- L’intestazione deve essere scritta in inglese, come già indicato, e non contenere logica esecutiva.

Regole di organizzazione interna
- Imports e dipendenze devono essere isolati nel primo blocco.
- Configurazioni e path devono essere separati in uno o più blocchi dedicati.
- Le utility devono essere raggruppate in blocchi autonomi.
- La logica principale deve essere separata dalla fase di esecuzione (main flow).
- Il blocco di esecuzione principale (main) deve essere sempre posizionato alla fine dello script
  e deve corrispondere all’ultimo blocco numerato.
- L’ordine dei blocchi deve seguire il flusso logico dello script.

Standard di funzioni
- Ogni funzione deve includere una descrizione inline su singola riga.
- La descrizione deve essere breve e in inglese.
- Il formato deve essere:
  nome_funzione – descrizione breve (one line)
- Le funzioni devono essere logicamente raggruppate per area funzionale.

Riferimento ufficiale
- Tutti gli script devono seguire il template definito in:
  /docs/templates/script_structure_template.txt
- Il template script_structure_template.txt rappresenta il modello strutturale ufficiale
  e l’implementazione canonica delle regole definite in questa sezione.
  
Gestione messaggistica (Python + Runtime system)
------------------------------------------------------------
- Tutti i messaggi di interfaccia Python devono essere gestiti tramite:
  core/messages.py → classe Messages
- Il sistema centralizza:
  - caricamento lingua da settings.json (interface_lang)
  - lettura file JSON da locale/Active/
  - In caso di chiavi mancanti nei file JSON, il sistema utilizza fallback esplicito [MISSING: key].
- Lo script principale e tutti i moduli runtime devono utilizzare esclusivamente questo sistema.
- Eccezione:
  script di bootstrap iniziale e script di servizio possono usare output non localizzato.

Gestione lingue (locale)
------------------------------------------------------------
- La gestione delle lingue è centralizzata nella cartella locale/
- locale/Active/
  contiene solo lingue localizzate e attive nel sistema
  → usate per interfaccia e runtime
- locale/System/
  contiene metadati e definizioni di sistema:
  - languages.json → lingue concettualmente supportate
  - whisper_languages.json → lingue effettivamente supportate da Whisper
- Le lingue attive sono l’unico set valido per menu e selezione utente.

Gestione messaggistica PowerShell
------------------------------------------------------------
- Sistema basato su moduli dedicati:
  ps/Messages.psm1
  ps/Logging.psm1
- Messages.psm1:
  - gestione messaggi tramite chiavi
  - supporto Hashtable e JSON object
  - interpolazione parametri
  - fallback: "Missing key: <key>"
- Logging.psm1:
  - Write-Log
  - livelli: INFO / WARN / ERROR
  - integrazione con Get-Message
  - output console colorato per livello
- Il sistema PowerShell è equivalente concettualmente al sistema Python,
  ma separato per ambiente runtime.
- I moduli PowerShell devono utilizzare esclusivamente questo sistema.
- I moduli legacy non ancora allineati sono considerati compatibilità temporanea.

Script di test
------------------------------------------------------------
- Tutti gli script di test devono avere prefisso obbligatorio:
  test_
- Il prefisso deve essere rimosso quando lo script entra nel flusso operativo.
- Non sono ammessi nomi alternativi.

Compatibilità legacy
------------------------------------------------------------
- Il codice esistente che non rispetta le convenzioni viene mantenuto invariato.
- Le nuove regole NON devono rompere la compatibilità con il codice già funzionante.
- Il refactoring completo del codice legacy verrà pianificato separatamente,
  solo dopo la stabilizzazione della pipeline principale.

Code Documentation Rules
------------------------------------------------------------
- I commenti nel codice devono essere scritti in inglese.
- Le stringhe dell’interfaccia utente devono essere gestite tramite sistema di localizzazione.
- Durante lo sviluppo si utilizza italiano come base (locale/Active/it.json),
  ma il codice deve rimanere documentato in inglese.
- Le stringhe di interfaccia possono essere sviluppate in italiano come base locale,
  ma devono essere sempre strutturate per successiva internazionalizzazione.
- L’obiettivo è mantenere un codice internazionalmente leggibile indipendentemente dalla lingua dell’interfaccia.

Gestione File di Lavoro e File Info
------------------------------------------------------------
- Ogni script che genera file crea una sottocartella nella directory di destinazione:
  (Audio_Extracted/, Transcripts/, Translated/)
  formato: <timestamp>_<nome_file_di_partenza>
- In ogni sottocartella viene creato:
  <nome_file_di_partenza>_info.txt

- Contenuto minimo file info:
  - File di origine (percorso completo)
  - Data e ora creazione
  - Numero file/tracce generate

- Possibili informazioni aggiuntive:
  - Durata
  - Frequenza campionamento
  - Bitrate
  - Formato
  - Note

- I file in Video_Input/ non vengono mai modificati.
- Tutti i file generati restano nelle sottocartelle e sono tracciati tramite file info.
- Questa convenzione è obbligatoria per garantire tracciabilità e gestione ordinata dei dati.

- Refactoring futuro previsto: valutazione di un modello “project-based”
  in cui tutti i file (audio, trascrizioni, traduzioni) vengono raggruppati
  in una singola cartella per progetto.

- In questo modello ogni progetto avrebbe:
  - una directory dedicata per input/output
  - un unico file info centralizzato aggiornato durante le fasi della pipeline

- L’obiettivo del refactoring è migliorare:
  - tracciabilità end-to-end
  - isolamento tra progetti paralleli
  - riduzione della dispersione dei file tra directory globali

- La modifica non è parte dell’implementazione attuale e verrà valutata
  solo dopo stabilizzazione completa della pipeline corrente.
- Questa evoluzione è classificata come refactoring architetturale futuro e non influisce sul comportamento attuale.

- Il recapDubbing è uno strumento di supporto alla comprensione del codice e non fa parte del contratto strutturale del progetto.
- Le linee guida non sono vincolanti, ma sono consigliate per garantire coerenza e leggibilità tra le overview degli script.

- Overview script (opzionale)
------------------------------
- È possibile includere una overview per fornire una sintesi chiara dello scopo e del comportamento dello script.
- È particolarmente utile per moduli core o complessi.
- Gli script semplici o di servizio possono ometterla.

- Formato consigliato (non vincolante)
--------------------------------------
- Nome del modulo
- Funzione generale (descrizione breve)
- Principali passaggi operativi (flusso di esecuzione)
- Funzioni principali (nome + parametri in formato compatto)
- Note aggiuntive (facoltative)

StartDubbing.bat
- Script di avvio del progetto DubbingToolkit.
- Si posiziona automaticamente nella cartella in cui si trova (indipendentemente dal disco).
- Controlla la presenza di PowerShell (Windows 5.1 o 7) e segnala errore se non disponibile.
- Imposta il percorso locale di ffmpeg (Tools\ffmpeg-7.1.1-full_build\bin) e lo aggiunge temporaneamente al PATH di sistema.
- Avvia Launcher.ps1 con policy bypass e finestra PowerShell dedicata, mantenendo la sessione aperta per eventuali controlli.
- Lista funzioni principali:
	nessuna funzione interna, script eseguibile batch standalone.
- Note aggiuntive:
	L’avvio del progetto dipende dal corretto posizionamento della cartella Tools e del file Launcher.ps1.
	Messaggi di log informativi tramite echo direttamente in console.

Launcher.ps1
- Script di bootstrap del progetto.
- Imposta encoding UTF-8 e code page per output corretto.
- Gestisce la creazione dei log di avvio (Logs/), con rotazione ultimi 10 log.
- Gestisce la selezione dinamica della lingua leggendo Locale/System/languages.json e i file lingua disponibili in Locale/Active.
- Carica i messaggi localizzati (<lang>.json) e li usa per tutto l’output successivo.
- Imposta la lingua selezionata come variabile d’ambiente SELECTED_LANG.
- Verifica presenza di Python, scarica e installa se necessario.
- Avvia e gestisce:
  • VenvManager.ps1 (creazione/attivazione venv)
  • InstallDependencies.ps1 (dipendenze Python)
- Avvia Regista.py usando il Python del virtual environment.
- Fornisce checkpoint informativi e messaggi localizzati durante tutto il flusso.

VenvManager.ps1
- Gestisce il virtual environment Python del progetto.
- Legge la lingua dall’ambiente (SELECTED_LANG) per caricare i messaggi localizzati.
- Verifica l’esistenza e l’integrità del venv (controllo struttura e file core).
- Se mancante o corrotto, elimina e ricrea il virtual environment.
- Attiva il venv nella sessione PowerShell corrente.
- Se il venv è stato appena creato, genera il file whisper_languages.json tramite script Python dedicato.

InstallDependencies.ps1
- Gestisce l’installazione delle dipendenze Python del progetto.
- Legge la lingua dall’ambiente (SELECTED_LANG) per caricare i messaggi localizzati.
- Interroga i pacchetti già installati nel venv.
- Confronta requirements.txt con le dipendenze presenti nel venv.
- Installa solo i pacchetti mancanti tramite pip.
- Verifica l’installazione di ogni pacchetto prima di proseguire.


Regista.py
- Script principale di orchestrazione del progetto.
- Legge la lingua dall’ambiente (SELECTED_LANG) e carica i messaggi localizzati (fallback su inglese).
- Gestisce il menu principale e il flusso di lavoro, inclusi: estrazione tracce, trascrizione, traduzione, doppiaggio e gestione sottotitoli.
- Gestisce le impostazioni iniziali e il file reset.json per rilevare il primo avvio e applicare le configurazioni di default.
- Seleziona il motore TTS e coordina l’esecuzione dei moduli Python collegati.
- Fornisce messaggi localizzati e logica interattiva all’utente tramite console.

estrai_tracce.py
- Script per l’estrazione delle tracce audio dai file video
- Permette di selezionare il video da cartella Video_Input o tramite browse manuale
- Conta automaticamente il numero di tracce audio presenti nel video
- Crea una cartella timestampata in Audio_Extracted per salvare le tracce
- Estrae le tracce audio in formato WAV mantenendo codec, sample rate e canali originali
- Genera un file di riferimento .info, nome formato <video_name>_info.txt, contenente: file video di origine, cartella estrazione, data estrazione, numero di tracce, dimensione totale, nome traccia, durata, dimensione, formato, sample rate, canali, profondità bit, codec
- Gestisce errori e avvisi con messaggi localizzati tramite file JSON
- Aggiorna lo storico estrazioni usando manage_history
- Può essere eseguito stand-alone per test rapido caricando i messaggi localizzati

trascrivi_audio.py
- Script per la trascrizione automatica delle tracce audio estratte.
- Permette di selezionare il file audio da Audio_Extracted o tramite browse manuale.
- Richiede conferma all’utente prima di avviare la trascrizione, con default “Sì” se si preme Enter senza inserire nulla.
- Determina automaticamente il dispositivo di elaborazione disponibile: GPU NVIDIA con CUDA o CPU; supporto a schede AMD non testato.
- Utilizza Whisper (modello selezionabile: tiny, base, small, medium, large) per generare la trascrizione; rileva automaticamente la lingua parlata e la indica nei file info.
- Confronta lingua rilevata e lingua impostata nelle impostazioni, avvisando l’utente in caso di discrepanza e chiedendo conferma per procedere.
- Crea una cartella timestampata nella sottocartella corrispondente a Transcripts/ per salvare i file generati.
- Genera i seguenti file SRT:
	<audio_name>_<lang>_raw.srt → trascrizione raw direttamente da Whisper
	<audio_name>_<lang>_seg.srt → trascrizione segmentata opzionale basata sulle pause
	<audio_name>_<lang>_work.srt → file per editing manuale (copia di seg o raw)
Genera un file info tramite InfoManager (<audio_name>_info.txt) con contenuto minimo:
	File audio di origine
	Cartella di destinazione
	Data/ora creazione
	Lingua parlata in forma leggibile (es. “italiano”)
	Modello Whisper usato
	Dispositivo utilizzato (CPU o GPU)
	File SRT generati (raw, seg, work)
	Numero di segmenti trascritti
	Durata audio in secondi
	Stato segmentazione abilitata o disabilitata
- Gestisce eventuali errori e avvisi durante la trascrizione con messaggi localizzati tramite JSON.
- Mantiene uno storico delle trascrizioni tramite manage_history.
- Può essere eseguito stand-alone per test rapido caricando i messaggi localizzati.

backup_utils.py
- Script per la gestione e manutenzione dei backup di progetto.
- Controlla l’esistenza della cartella di lavoro e segnala errori se mancante.
- Lista tutte le sottocartelle presenti e segnala se non ci sono backup disponibili.
- Raggruppa le cartelle per lingua in base al nome della cartella.
- Applica il limite massimo di backup per lingua, eliminando i più vecchi (opzione cestino o cancellazione definitiva).
- Applica il limite totale di backup, rimuovendo le cartelle più vecchie fino a rispettare il numero massimo consentito.
- Se impostato, verifica la dimensione totale dei backup e segnala un warning se supera il limite configurato.
- In caso di superamento dimensione, chiede all’utente se procedere con la pulizia.
- Permette all’utente di scegliere il metodo di cancellazione (cestino o definitivo) per liberare spazio.
- Rimuove le cartelle più vecchie fino a rientrare nel limite di dimensione, aggiornando in tempo reale il calcolo della dimensione.
-Tutte le operazioni di rimozione o errori vengono loggate con messaggi colorati tramite i file JSON dei messaggi.

settings_manager.py
- Gestisce la lettura, modifica e reset delle impostazioni del progetto.
- Mostra un menu interattivo con opzioni per modificare le impostazioni, soft reset, cambio lingua e hard reset.
- Permette di modificare il numero massimo di backup totali e per lingua.
- Permette di modificare la dimensione massima delle cartelle di backup in GB.
- Permette di attivare o disattivare le conferme e l’uso del cestino per la cancellazione dei backup.
- Salva le impostazioni aggiornate nel file settings.json.
- Gestisce la selezione della lingua dell’interfaccia mostrando le lingue disponibili e salvando la scelta.
- Aggiorna i messaggi dell’interfaccia in base alla lingua selezionata.
- Consente di ripristinare le impostazioni di default tramite soft reset.
- Consente di eseguire un hard reset impostando flag di reset e lanciando lo script reset_env.ps1.
- Gestisce conferme multiple per evitare reset accidentali.
- Fornisce messaggi di conferma o errore in console con output colorato.

monitoraggio_consumo.py

info_manager.py
- Classe InfoManager per gestione sicura di JSON e file info leggibili per ogni progetto.

Funzionalità principali:
- Caricamento e salvataggio JSON (`save_json`) con aggiornamento timestamp ultima modifica.
- Aggiornamento sezione progetto (`update_project`) con impostazioni e lingue da settings.
- Aggiornamento file video (`update_video_file`) e analisi opzionale con ffprobe.
- Aggiunta tracce audio (`add_audio_track`) con rilevamento proprietà WAV o tramite mutagen/ffprobe per altri formati.

    - Durata audio

    - Sezioni: Progetto, Video, Audio, Trascrizione, Traduzioni, TTS

- Aggiornamento TTS (`update_tts`) con caratteri usati, franchigia e costo stimato.

- Gestione sicura dell’ereditarietà dai JSON precedenti (`_inherit_previous`) senza sovrascrivere la sezione corrente.
- Logging degli errori e warning per caricamento, salvataggio e analisi file.
Uso tipico:

- Permette una visione centralizzata e leggibile dello stato di progetto e delle risorse multimediali.

utils_tts.py
- Modulo di funzioni generiche per supporto TTS, gestione file e parsing SRT.
- Lista funzioni principali:
	parse_srt(file_path: str) -> list[tuple]
		- Legge un file SRT e restituisce una lista di tuple (filename, testo, start_time, end_time)
		- Il filename è generato come "HH-MM-SS-ms_HH-MM-SS-ms" corrispondente a inizio e fine frase
		- Divide i blocchi SRT separati da righe vuote e estrae timestamp e testo
	genera_nome_file(timestamp: str, estensione: str = "wav") -> str
		- Genera il nome del file audio a partire dal timestamp
		- Restituisce stringa tipo "HH-MM-SS-ms_HH-MM-SS-ms.wav"
	assicurati_cartella(path: str) -> str
		- Controlla se la cartella esiste; se no, la crea
		- Restituisce il percorso della cartella
- Note aggiuntive:
	- Nessuna dipendenza da funzioni di sintesi esterne
	- Tutte le funzioni sono standalone e facilmente integrabili in script TTS specifici
	- La funzione `sintetizza_batch` è presente solo come commento e non viene utilizzata

================== Analisi PROGETTO TTS DUBBING ==================

1. tts_dubbing.py
   - Orchestratore principale per TTS batch.
   - Input: file SRT + config (provider, voce, lingua, formato).
   - Gestisce:
     • Selezione provider (Google/Azure)
     • Sintesi vocale batch (chiama rispettivamente tts_google/tts_azure)
     • Calcolo consumo caratteri (ConsumoTTS)
     • Merge file audio (tts_merge)
   - Output: file unico audio nella cartella Dubbed/<Provider>/<file_stem>.

2. monitoraggio_consumo.py
   - Classe ConsumoTTS: gestisce storico consumo caratteri TTS.
   - Tiene traccia di franchigia, costo stimato, percentuali.
   - Lock per scrittura sicura su JSON multithread.
   - Funzioni utili: carica, salva, aggiungi_caratteri, report_sintesi, report_completo, stampa_storico, pulisci_mesi_vecchi.

3. tts_merge.py
   - Funzione merge_audio_files: concatena file audio generati in un unico file.
   - Supporta formati wav e mp3.
   - CLI standalone per unire file da cartella input.

4. tts_config_manager.py
   - Gestione config TTS e stato corrente (file, lingua, provider, voce, output_format).
   - Funzioni:
     • aggiorna_file_srt() → seleziona nuovo file SRT, resetta config
     • aggiorna_lingua() → legge da project_info.json o menu manuale
     • aggiorna_motore_tts() → selezione provider (Google/Azure)
     • aggiorna_voce() → seleziona voce disponibile
     • aggiorna_formato_output() → seleziona mp3/wav

5. core/utils_tts.py
   - Funzioni generiche TTS:
     • parse_srt(file) → lista blocchi (filename, testo, start_time, end_time)
     • genera_nome_file(timestamp, estensione)
     • assicurati_cartella(path)
     • count_characters(file) → numero totale caratteri
   - (opzionale) sintetizza_batch generica per provider

6. tts_google.py
   - Sintesi vocale batch con Google TTS.
   - Parsing SRT interno, creazione client Google.
   - Ciclo su entry: genera file output, chiama client.synthesize_speech.
   - Supporta output mp3/wav.
   - CLI standalone disponibile.

7. tts_azure.py
   - Sintesi vocale batch con Azure TTS.
   - Parsing SRT interno, creazione SpeechConfig con voce e formato.
   - Ciclo su entry SRT: genera file output, chiama SpeechSynthesizer.speak_text_async.
   - Supporta output mp3/wav.
   - CLI standalone disponibile.

================== FLUSSO LAVORO ==================
1. Selezione file SRT → tts_config_manager
2. Configurazione lingua, provider, voce, output → tts_config_manager
3. Chiamata tts_dubbing(config):
   • Calcolo caratteri → ConsumoTTS
   • Sintesi batch → tts_google/tts_azure
   • Merge file → tts_merge
4. Storico consumo → monitoraggio_consumo.py
5. Output finale → file unico audio in Dubbed/<Provider>/<SRT_name>.<formato>
