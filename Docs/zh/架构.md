# 架构与技术参考

本文档描述项目的内部结构、主要模块、开发规范及组件状态，主要面向参与开发或希望了解系统内部运行机制的人员。

---

## 目录结构

```
DubbingToolkit/
├── Audio_Extracted/        音频提取输出（按文件创建子目录）
├── Audio_Input/            直接音频输入
├── Billing/                TTS 消耗与费用监控
├── core/                   共享的 Python 支持模块
├── credentials/            API 凭据（不纳入 Git 管理）
├── Dubbed/                 TTS 最终输出（按提供商分类）
├── Installation/           本地 Python 运行环境（3.10、3.11）
├── installer/              构建与打包系统
├── locale/                 本地化与语言管理
│   ├── Active/             活动 JSON 语言文件（it、en、es、de、fr、pt、ru、zh）
│   └── System/             语言元数据（Whisper、支持的语言）
├── Logs/                   操作日志
├── ps/                     PowerShell 模块（日志、消息）
├── Repository/             共享资源与本地模型
├── Scripts/                操作脚本与 Python 模块
│   └── maintenance/        维护脚本与本地化流程
├── Settings/               活动配置与参考配置
├── Subtitles/              字幕（待实现）
├── Temp/                   临时文件
├── Tools/                  外部可执行文件（ffmpeg）
├── Transcripts/            SRT 转录文件（按文件创建子目录）
├── Translated/             SRT 翻译文件（按文件创建子目录）
├── venv/                   主 Python 虚拟环境
├── Video_Input/            源视频文件（从不修改）
└── voices/                 可用 TTS 语音及音频示例
```

## 启动链

```
StartDubbing.bat
  └→ Scripts/Launcher.ps1
       激活 venv、配置 UTF-8、日志、加载语言
         └→ Scripts/Regista.py
              主菜单与流程编排
```

Launcher 负责：选择本地 Python 运行环境（`Installation/`）、创建/激活 venv、配置日志系统、加载界面语言。

`Regista.py` 是中心协调器：向用户显示菜单，并将执行委托给各阶段的专用模块。

---

## 操作流程

| 阶段 | 模块 | 输入 → 输出 |
|---|---|---|
| 1 — 音频提取 | `Scripts/estrai_tracce.py` | `Video_Input/` → `Audio_Extracted/<ts>/` |
| 2 — 转录 | `Scripts/trascrivi_audio.py` | `Audio_Extracted/` 或 `Audio_Input/` → `Transcripts/<ts>/`（SRT） |
| 3 — 翻译 | `Scripts/traduci_testo.py` | `Transcripts/` → `Translated/<ts>/`（SRT） |
| 4 — TTS | `Scripts/tts_menu.py` | `Translated/` → `Dubbed/`（MP3/WAV） |

`tts_menu.py` 根据活动提供商委托给 `tts_azure.py` 或 `tts_google.py`。

## 核心模块（`core/`）

这些模块被整个流程共享，不应由用户直接调用。

### `messages.py`
本地化消息的集中系统。读取 `Settings/settings.json` → `interface_lang` 字段 → 加载 `locale/Active/<lang>.json`。

脚本中的使用方式：
```python
from core.messages import Messages
msg = Messages()
print(msg._("消息键"))
```

缺失的键会产生 `[MISSING: 键]` 回退，不会导致崩溃。所有缺失键必须在发布前修复。

### `credentials_manager.py`
API 凭据的集中加载与验证。这是项目中唯一有权读取 `credentials/` 中文件的位置，其他模块不得直接访问这些文件。

### `api_check.py`
在允许访问 TTS 菜单前验证凭据的有效性，进入 TTS 菜单时自动调用。

### `ui_printer.py` + `ui_colors.py`
带格式和颜色的控制台输出函数。所有脚本必须使用这些模块而非直接使用 `print()`，以确保视觉一致性。

### `utils_tts.py`
两个 TTS 后端共用的 SRT 解析工具。

### `file_selector.py`
通过交互式菜单选择文件的界面。

### `input_parsing.py`
用户输入的解析与验证。

## 主要 Scripts 模块

### `Regista.py`
主编排器。管理顶层菜单并协调流程各阶段的执行，是应用程序的 Python 入口点。

### `estrai_tracce.py`
通过 ffmpeg 从视频中提取音轨。在 `Audio_Extracted/` 中生成包含音频文件和 `_info.txt` 的子目录。

### `trascrivi_audio.py`
通过 Whisper（或集成后的 WhisperX）进行音频转录，输出 SRT 格式到 `Transcripts/`。

### `traduci_testo.py`
通过本地运行的 Helsinki-NLP MarianMT 模型翻译 SRT，输出到 `Translated/`。

### `tts_dubbing.py` / `tts_menu.py`
TTS 流程协调。`tts_menu.py` 是用户界面；`tts_dubbing.py` 管理段落生成与合并流程。

### `tts_azure.py` / `tts_google.py`
提供商专用 TTS 后端，生成音频段落并保存到 `Dubbed/<PROVIDER>/`。

### `tts_merge.py`
将 TTS 音频段落合并同步为最终文件。

### `tts_config_manager.py`
管理 TTS 配置：活动提供商、选定语音、合成参数。

### `info_manager.py`
读写带时间戳子目录中的 `project_info.json` 文件，确保各阶段之间的状态可追溯性。

### `settings_manager.py`
读取、验证和访问 `Settings/settings.json` 中的配置。

### `monitoraggio_consumo.py`
线程安全地访问 `Billing/consumo_tts.json` 中的 TTS 消耗记录。

### `menu_lingue.py` / `menu_lingue_tts.py`
为转录/翻译及 TTS 流程选择语言。

### `menu_voices.py`
通过界面选择和配置 TTS 语音。

### `backup_utils.py`
管理生成文件的备份和历史记录。

## 本地化系统

### 结构

```
locale/
├── Active/              活动语言文件（运行时）
│   ├── it.json
│   ├── en.json
│   ├── es.json
│   ├── de.json
│   ├── fr.json
│   ├── pt.json
│   ├── ru.json
│   └── zh.json
└── System/
    ├── languages.json           概念上支持的语言
    └── whisper_languages.json   Whisper 支持的语言
```

### 规则

- 所有 Python 界面消息必须使用 `core/messages.py`。
- `locale/Active/` 中的所有文件必须保持同步：`it.json` 中存在的键必须存在于所有其他语言文件中。
- 缺失的键在运行时会产生 `[MISSING: key]`——在稳定环境中不允许出现。
- PowerShell 使用 `ps/Messages.psm1`（等效的独立系统）。

### 本地化维护流程

在 `Scripts/maintenance/translation/` 中提供了一套完整的语言文件管理流程：

| 脚本 | 功能 |
|---|---|
| `LocaleKeyAnalyzer.ps1` | 分析文件间缺失键和不一致性 |
| `LocaleTranslator.ps1` | 带占位符保护的自动翻译 |
| `Validate-LocaleJson.ps1` | 验证 JSON 结构和完整性 |
| `Fix-LocaleDuplicates.ps1` | 修复重复键 |
| `Clean-LocaleUnusedKeys.ps1` | 删除未使用的键 |
| `Extract-Placeholders.ps1` | 提取和映射占位符 |
| `Protect-Placeholders.ps1` | 自动翻译期间保护占位符 |

## 配置（`Settings/settings.json`）

主要字段：

```json
{
  "interface_lang": "it",
  "model": "small",
  "Transcript_Audio_Spoken_Lang": "it",
  "Translation_Target_Lang": "en",
  "Dubbing_Lang": "en"
}
```

| 字段 | 描述 |
|---|---|
| `interface_lang` | 用户界面语言 |
| `model` | 使用的 Whisper 模型（`tiny`、`base`、`small`、`medium`、`large`） |
| `Transcript_Audio_Spoken_Lang` | 源音频中的口语语言 |
| `Translation_Target_Lang` | 翻译目标语言 |
| `Dubbing_Lang` | TTS 合成语言 |

## 构建与发布系统

项目在 `installer/` 中包含打包系统：

```powershell
installer/build.ps1
```

包含/排除规则位于：

| 文件 | 用途 |
|---|---|
| `build_include.json` | 复制什么、复制到哪里、以何种方式 |
| `build_exclude.json` | 全局黑名单（所有模式） |
| `build_exclude_test.json` | 仅 TEST 模式的额外黑名单（Python 运行时、ffmpeg、voices） |
| `build_protected.json` | 逐字复制的路径——排除规则被忽略 |
| `build_empty_dirs.json` | 在包中创建的空目录 |

可用参数：
```powershell
.\build.ps1              # 交互式菜单（选择 1/2/3）
.\build.ps1 -Test        # 不含重型组件的轻量构建
.\build.ps1 -Production  # 带确认的完整构建
.\build.ps1 -DryRun      # 不写入文件的模拟运行
```

输出到 `installer/build_payload/`。真实凭据文件绝不包含在构建中——仅包含 `*.template.json` 文件。

## 开发规范

### 命名

| 元素 | 规范 |
|---|---|
| 目录（新建） | `小写_下划线` |
| Python 模块 | `小写_下划线.py` |
| 类 | `CamelCase` |
| 函数和变量 | `小写_下划线` |
| 测试脚本 | 必须以 `test_` 为前缀 |

### 脚本结构

所有脚本必须遵循 `RecapDubbing.txt` 第 6 节中定义的编号结构：

```
# 1. IMPORTS / DEPENDENCIES
# 2. CONFIGURATION – Paths, settings, constants
# 3. UTILITIES – Helper functions
# 4. CORE LOGIC – Main processing
# 5. MAIN EXECUTION – Entry point
```

每个脚本必须包含标准头部，包含名称、描述、输入、输出和操作说明。代码注释必须使用英文。

### 消息

- 运行时模块中不得有硬编码字符串。
- 所有消息来自本地化 JSON 文件，通过 `core/messages.py`（Python）或 `ps/Messages.psm1`（PowerShell）。
- 例外：引导脚本和维护脚本可以使用非本地化输出。

### 路径

所有路径必须相对于项目根目录。运行时模块中不得使用绝对路径。

### 生成的文件

每个生成文件的流程阶段都会创建一个格式为以下的子目录：
```
<时间戳>_<源文件名>
```
并包含一个带有处理元数据的 `_info.txt` 文件。
