# Архитектура и техническое описание

Этот документ описывает внутреннюю структуру проекта, основные модули, соглашения о разработке и состояние компонентов. Предназначен прежде всего для тех, кто участвует в разработке или хочет понять внутреннее устройство системы.

---

## Структура папок

```
DubbingToolkit/
├── Audio_Extracted/        Выходные файлы извлечения аудио (подпапки по файлам)
├── Audio_Input/            Прямой входной аудиофайл
├── Billing/                Мониторинг потребления и затрат TTS
├── core/                   Общие вспомогательные модули Python
├── credentials/            Учётные данные API (исключены из Git)
├── Dubbed/                 Окончательный выход TTS (по провайдеру)
├── Installation/           Локальные среды выполнения Python (3.10, 3.11)
├── installer/              Система сборки и упаковки
├── locale/                 Локализация и управление языками
│   ├── Active/             Активные JSON-файлы языков (it, en, es, de, fr, pt, ru, zh)
│   └── System/             Метаданные языков (Whisper, поддерживаемые языки)
├── Logs/                   Операционные журналы
├── ps/                     Модули PowerShell (логирование, сообщения)
├── Repository/             Общие ресурсы и локальные модели
├── Scripts/                Операционные скрипты и модули Python
│   └── maintenance/        Скрипты обслуживания и конвейер локализации
├── Settings/               Активная конфигурация и справочная
├── Subtitles/              Субтитры (предстоит реализовать)
├── Temp/                   Временные файлы
├── Tools/                  Внешние исполняемые файлы (ffmpeg)
├── Transcripts/            Транскрипции SRT (подпапки по файлам)
├── Translated/             Переводы SRT (подпапки по файлам)
├── venv/                   Основная виртуальная среда Python
├── Video_Input/            Исходные видеофайлы (никогда не изменяются)
└── voices/                 Доступные голоса TTS и аудиообразцы
```

## Цепочка запуска

```
StartDubbing.bat
  └→ Scripts/Launcher.ps1
       Активирует venv, настройка UTF-8, журнал, загрузка языка
         └→ Scripts/Regista.py
              Главное меню и оркестрация конвейера
```

Launcher управляет: выбором локальной среды выполнения Python (`Installation/`), созданием/активацией venv, настройкой системы журналирования, загрузкой языка интерфейса.

`Regista.py` — центральный координатор: отображает меню пользователю и делегирует выполнение конкретным модулям для каждого этапа.

## Операционный конвейер

| Этап | Модуль | Вход → Выход |
|---|---|---|
| 1 — Извлечение аудио | `Scripts/estrai_tracce.py` | `Video_Input/` → `Audio_Extracted/<ts>/` |
| 2 — Транскрипция | `Scripts/trascrivi_audio.py` | `Audio_Extracted/` или `Audio_Input/` → `Transcripts/<ts>/` (SRT) |
| 3 — Перевод | `Scripts/traduci_testo.py` | `Transcripts/` → `Translated/<ts>/` (SRT) |
| 4 — TTS | `Scripts/tts_menu.py` | `Translated/` → `Dubbed/` (MP3/WAV) |

`tts_menu.py` делегирует к `tts_azure.py` или `tts_google.py` в зависимости от активного провайдера.

## Основные модули (`core/`)

Эти модули используются всем конвейером. Пользователь не должен вызывать их напрямую.

### `messages.py`
Централизованная система локализованных сообщений. Читает `Settings/settings.json` → поле `interface_lang` → загружает `locale/Active/<lang>.json`.

Использование в скриптах:
```python
from core.messages import Messages
msg = Messages()
print(msg._("ключ_сообщения"))
```

Отсутствующие ключи выдают заглушку `[MISSING: ключ]` и не вызывают сбоев. Все отсутствующие ключи должны быть исправлены до выпуска.

### `credentials_manager.py`
Централизованная загрузка и проверка учётных данных API. Единственная точка проекта, уполномоченная читать файлы в `credentials/`. Никакой другой модуль не должен обращаться к этим файлам напрямую.

### `api_check.py`
Проверяет действительность учётных данных перед разрешением доступа к меню TTS. Вызывается автоматически при входе в меню TTS.

### `ui_printer.py` + `ui_colors.py`
Функции для вывода в консоль с форматированием и цветами. Все скрипты должны использовать эти модули вместо прямого `print()` для обеспечения визуальной согласованности.

### `utils_tts.py`
Общие утилиты для разбора SRT, используемые обоими бэкендами TTS.

### `file_selector.py`
Интерфейс для выбора файлов через интерактивное меню.

### `input_parsing.py`
Разбор и проверка пользовательского ввода.

## Основные модули Scripts

### `Regista.py`
Главный оркестратор. Управляет меню верхнего уровня и координирует выполнение этапов конвейера. Точка входа Python приложения.

### `estrai_tracce.py`
Извлечение аудиодорожек из видео через ffmpeg. Создаёт подпапку в `Audio_Extracted/` с аудиофайлами и файлом `_info.txt`.

### `trascrivi_audio.py`
Транскрипция аудио через Whisper (или WhisperX, когда интегрирован). Выход в формате SRT в `Transcripts/`.

### `traduci_testo.py`
Перевод SRT через локально запущенные модели Helsinki-NLP MarianMT. Выход в `Translated/`.

### `tts_dubbing.py` / `tts_menu.py`
Координация конвейера TTS. `tts_menu.py` — пользовательский интерфейс; `tts_dubbing.py` управляет потоком генерации и слияния сегментов.

### `tts_azure.py` / `tts_google.py`
Специфичные для провайдера бэкенды TTS. Генерируют аудиосегменты и сохраняют их в `Dubbed/<PROVIDER>/`.

### `tts_merge.py`
Слияние и синхронизация аудиосегментов TTS в итоговый файл.

### `tts_config_manager.py`
Управление конфигурациями TTS: активный провайдер, выбранный голос, параметры синтеза.

### `info_manager.py`
Чтение и запись файла `project_info.json` в подпапках с временными метками. Обеспечивает отслеживаемость состояния между этапами.

### `settings_manager.py`
Чтение, проверка и доступ к конфигурациям в `Settings/settings.json`.

### `monitoraggio_consumo.py`
Потокобезопасный доступ к журналу потребления TTS в `Billing/consumo_tts.json`.

### `menu_lingue.py` / `menu_lingue_tts.py`
Выбор языков для транскрипции/перевода и конвейера TTS.

### `menu_voices.py`
Выбор и настройка голосов TTS из интерфейса.

### `backup_utils.py`
Управление резервными копиями и историей сгенерированных файлов.

## Система локализации

### Структура

```
locale/
├── Active/              Активные языковые файлы (runtime)
│   ├── it.json
│   ├── en.json
│   ├── es.json
│   ├── de.json
│   ├── fr.json
│   ├── pt.json
│   ├── ru.json
│   └── zh.json
└── System/
    ├── languages.json           Концептуально поддерживаемые языки
    └── whisper_languages.json   Языки, поддерживаемые Whisper
```

### Правила

- Все сообщения интерфейса Python должны использовать `core/messages.py`.
- Все файлы в `locale/Active/` должны быть синхронизированы: ключ, присутствующий в `it.json`, должен существовать во всех других языковых файлах.
- Отсутствующие ключи выдают `[MISSING: key]` во время выполнения — не допускаются в стабильной среде.
- PowerShell использует `ps/Messages.psm1` (эквивалентная, отдельная система).

### Конвейер обслуживания локализации

В `Scripts/maintenance/translation/` доступен полный конвейер для управления языковыми файлами:

| Скрипт | Функция |
|---|---|
| `LocaleKeyAnalyzer.ps1` | Анализ отсутствующих ключей и несоответствий между файлами |
| `LocaleTranslator.ps1` | Автоматический перевод с защитой заполнителей |
| `Validate-LocaleJson.ps1` | Проверка структуры и целостности JSON |
| `Fix-LocaleDuplicates.ps1` | Исправление дублирующихся ключей |
| `Clean-LocaleUnusedKeys.ps1` | Удаление неиспользуемых ключей |
| `Extract-Placeholders.ps1` | Извлечение и сопоставление заполнителей |
| `Protect-Placeholders.ps1` | Защита заполнителей при автоматическом переводе |

## Конфигурация (`Settings/settings.json`)

Основные поля:

```json
{
  "interface_lang": "it",
  "model": "small",
  "Transcript_Audio_Spoken_Lang": "it",
  "Translation_Target_Lang": "en",
  "Dubbing_Lang": "en"
}
```

| Поле | Описание |
|---|---|
| `interface_lang` | Язык пользовательского интерфейса |
| `model` | Используемая модель Whisper (`tiny`, `base`, `small`, `medium`, `large`) |
| `Transcript_Audio_Spoken_Lang` | Язык речи в исходном аудио |
| `Translation_Target_Lang` | Целевой язык для перевода |
| `Dubbing_Lang` | Язык для синтеза TTS |

## Управление голосами TTS

Доступные голоса каталогизированы в `voices/`:

| Файл | Содержимое |
|---|---|
| `voices_azure.json` | Отфильтрованные голоса Azure, готовые к использованию |
| `voices_azure_complete.json` | Полный каталог Azure |
| `voices_google.json` | Отфильтрованные голоса Google |
| `voices_google_complete.json` | Полный каталог Google |
| `voices_index.json` | Единый индекс всех голосов (Azure + Google) с метаданными |

Аудиообразцы голосов (для прослушивания) находятся в `voices/voices_output/<provider>/<LANG-CODE>/<voice>.mp3`, если сгенерированы через `Scripts/VoicesRepository.py`.

Для обновления каталога голосов от провайдеров:
```bash
voices/fetch_azure_voices.py
voices/fetch_google_voices.py
```

## Система сборки и распространения

Проект включает систему упаковки в `installer/`:

```powershell
installer/build.ps1
```

Правила включения/исключения находятся в:

| Файл | Назначение |
|---|---|
| `build_include.json` | Что копировать, куда и в каком режиме |
| `build_exclude.json` | Глобальный чёрный список (все режимы) |
| `build_exclude_test.json` | Дополнительный чёрный список только в режиме TEST (Python runtime, ffmpeg, voices) |
| `build_protected.json` | Пути, копируемые дословно — правила исключения игнорируются |
| `build_empty_dirs.json` | Пустые папки для создания в пакете |

Доступные параметры:
```powershell
.\build.ps1              # интерактивное меню (выбор 1/2/3)
.\build.ps1 -Test        # лёгкая сборка без тяжёлых компонентов
.\build.ps1 -Production  # полная сборка с подтверждением
.\build.ps1 -DryRun      # симуляция без записи файлов
```

Выход идёт в `installer/build_payload/`. Реальные файлы учётных данных никогда не включаются в сборку — только файлы `*.template.json`.

## Соглашения о разработке

### Именование

| Элемент | Соглашение |
|---|---|
| Папки (новые) | `нижний_регистр_подчёркивание` |
| Модули Python | `нижний_регистр_подчёркивание.py` |
| Классы | `CamelCase` |
| Функции и переменные | `нижний_регистр_подчёркивание` |
| Тестовые скрипты | обязательный префикс `test_` |

### Структура скриптов

Все скрипты должны следовать нумерованной структуре, определённой в Разделе 6 `RecapDubbing.txt`:

```
# 1. IMPORTS / DEPENDENCIES
# 2. CONFIGURATION – Paths, settings, constants
# 3. UTILITIES – Helper functions
# 4. CORE LOGIC – Main processing
# 5. MAIN EXECUTION – Entry point
```

Каждый скрипт должен включать стандартный заголовок с именем, описанием, входными данными, выходными данными и операционными примечаниями. Комментарии в коде должны быть на английском.

### Сообщения

- Никаких жёстко закодированных строк в модулях runtime.
- Все сообщения поступают из JSON-файлов локализации через `core/messages.py` (Python) или `ps/Messages.psm1` (PowerShell).
- Исключение: скрипты начальной загрузки и скрипты обслуживания могут использовать нелокализованный вывод.

### Пути

Все пути должны быть относительными к корню проекта. Никаких абсолютных путей в модулях runtime.

### Сгенерированные файлы

Каждый этап конвейера, генерирующий файлы, создаёт подпапку в формате:
```
<временная_метка>_<имя_исходного_файла>
```
и включает файл `_info.txt` с метаданными обработки.
